### Day 2：3月20日（周五）—— 打透 Gemmini 数据流 + 乱序 CPU 基础

**Block 1：Gemmini 深度复盘（2h）**

- [ ] 从头到尾口述 Gemmini 数据流：
  ```
  CPU 发 RoCC 指令
  → LoopConv / LoopMatmul 拆解为 micro-ops
  → ReservationStation 调度（依赖追踪）
  → Load Controller: DRAM → Scratchpad (mvin)
  → Execute Controller: Scratchpad → Mesh → Accumulator (preload/compute)
  → Store Controller: Accumulator → DRAM (mvout)
  ```
- [ ] 准备以下关键问题的答案（每个 1-2 分钟口述）：
  - **为什么需要 Scratchpad 而不是用 cache？**
    - 确定性延迟、软件可控、避免 cache miss 抖动、适合 tiling 策略
  - **WS 和 OS 的核心差异？各适合什么场景？**
    - WS: 权重驻留 PE，输入流经阵列 → 适合大 batch / 权重复用高
    - OS: 输出驻留 PE，部分和原地累加 → 适合输出通道多
  - **为什么必须做 tiling？tiling 的决策因素是什么？**
    - Scratchpad 容量有限，大矩阵必须分块；决策因素：SP 大小、阵列维度、数据复用策略
  - **主要瓶颈是算力、带宽还是存储？**
    - 取决于算子特性；Conv3x3 通常 compute-bound；DWConv/1x1Conv 可能 memory-bound
  - **YOLOv11n 的算子划分依据是什么？**
    - 计算密集 + 规则数据访问 → 阵列；控制密集 / 非线性 / 不规则 → CPU

**Block 2：乱序 CPU 核心概念（1.5h）**

- [ ] 梳理乱序执行的 **完整流水线**：
  ```
  Fetch → Decode → Rename → Dispatch → Issue → Execute → Complete → Retire
  ```
- [ ] 搞清楚每个阶段做什么，为什么需要这个阶段
- [ ] 重点理解以下模块及其解决的问题：
  - **Tomasulo 算法**：解决 WAW/WAR 假依赖 + 实现动态调度
  - **寄存器重命名**：消除假依赖，扩大调度窗口
  - **ROB（Re-Order Buffer）**：保证顺序提交 + 支持精确异常 + 推测执行回滚

**Block 2：乱序 CPU 深入 + GPGPU 基础（2h）**

- [ ] 乱序 CPU 追问级准备（每题准备 1-2 分钟口述答案）：
  - "顺序核和乱序核在性能/面积/功耗/设计复杂度的权衡？"
    - 顺序核：简单、低功耗、低面积，适合 IoT/MCU；IPC 低
    - 乱序核：IPC 高但面积大、功耗高、验证复杂，适合高性能场景
  - "Tomasulo 和 Scoreboard 的区别？"
  - "ROB 为什么要顺序 retire？如果不顺序会怎样？"
    - 精确异常要求异常发生时之前的指令都已提交、之后的都没提交
  - "分支预测错了怎么恢复？"
    - Flush 错误路径指令、恢复 RAT（寄存器别名表）到 checkpoint、从正确目标重新取指
  - "如果让你给顺序核做性能增强，你优先加什么？"
    - 分支预测器（减少 stall）→ 数据前递（减少数据冒险 stall）→ 多发射
  - "RISC-V 的特点？和 ARM/x86 对比？"
    - 开源 ISA、模块化扩展（M/A/F/D/C/V + custom）、定长指令简化译码

- [ ] **GPGPU 基础认知**（这是 v1 完全缺失的，必须补）：
  - **SIMT 执行模型**：一条指令驱动多个线程（warp = 32 threads），区别于 SIMD
  - **Warp 调度**：多个 warp 交替执行来隐藏延迟（latency hiding）
  - **存储层次**：Register File → Shared Memory（类似 Scratchpad）→ L1/L2 Cache → Global Memory（DRAM）
  - **SM（Streaming Multiprocessor）结构**：多个 warp scheduler、多组 FP/INT ALU、SFU、Load/Store Unit、Shared Memory
  - **与 CPU 乱序核的对比**：
    - CPU OoO：少数线程、深流水线、复杂调度 → 优化单线程延迟
    - GPU：大量线程、宽 SIMT、简单调度 → 优化总吞吐量
  - **和你项目的连接**：脉动阵列的 Scratchpad ≈ GPU 的 Shared Memory（都是软件管理的片上存储）

**Block 3：AXI 协议复习（1h）**

- [ ] 五通道复习：AW / W / B / AR / R
  - 写有 5 通道因为写数据(W)和写地址(AW)可以解耦，且需要写响应(B)
  - 读只有 2 通道因为读数据和读响应可以合并在 R 通道
- [ ] Outstanding 传输：Master 可以不等前一笔完成就发起下一笔
- [ ] Backpressure：valid/ready 握手，ready 拉低实现反压
- [ ] Burst 类型：FIXED（FIFO 访问）/ INCR（连续地址）/ WRAP（cache line fill）
- [ ] 4KB 边界：AXI 不允许 burst 跨 4KB，因为 4KB 是最小页大小，跨页可能跨设备
- [ ] wstrb：按字节使能写入，支持非对齐写

**Block 4：异步 FIFO 原理梳理（0.5h）**

- [ ] 画异步 FIFO 框图：双时钟域、Gray code 指针、同步器、满/空判断
- [ ] 准备口述答案：

**Block 2：Transformer / MoE 硬件映射专项（1.5h）**

- [ ] **Attention 深入**：
  - Q = X × W_Q，K = X × W_K，V = X × W_V → 三次 MatMul，适合阵列
  - Attention Score = Q × K^T / √d_k → MatMul（阵列）+ 标量除法
  - Softmax(Score) → 行级 max → 减法 → exp → sum → div → **向量单元 / 专用电路**
  - Output = Softmax × V → MatMul（阵列）
  - **关键洞察**：Attention 中 ~80% 计算量在 MatMul，但 Softmax 是串行瓶颈
  - **Flash Attention 简要理解**：通过 tiling 和在线 softmax 减少 HBM 访问，计算换带宽

- [ ] **MoE 架构与硬件挑战**：
  - Router 根据输入 token 选择 top-k experts
  - 硬件挑战：
    - **负载不均衡**：不同 expert 被选中次数不同 → 计算资源利用不均
    - **数据通信**：token 需要路由到对应 expert → 跨核/跨芯片 All-to-All 通信
    - **动态调度**：expert 选择是运行时决定的 → 硬件需要灵活调度
  - 如果做硬件支持，需要：路由逻辑、跨核互联（Crossbar/NoC）、动态负载均衡

- [ ] **矩阵乘硬件实现与性能估计**（面经高频题）：
  - 脉动阵列实现 N×N 矩阵乘：N² 个 PE，每个 PE 做 MAC
  - 吞吐量：N² MACs/cycle
  - 延迟：约 3N cycles（数据流入 + 计算 + 流出）
  - 性能估计例题：16×16 阵列 @500MHz，峰值 = 16×16×2×500M = 256 GOPS

**Block 2：CDC + 异步 FIFO 理论（1h）**

- [ ] CDC 核心概念：
  - 单比特信号：双触发器同步器（2FF synchronizer）
  - 为什么需要两级 FF？→ 第一级可能亚稳态，第二级等待稳定
  - 多比特信号为什么不能逐位同步？→ 各位到达时间不同，可能采到中间态
  - 多比特方案：Gray code + 同步器（异步 FIFO）、握手协议、DMUX 方案
- [ ] 异步 FIFO 的 CDC 原理串讲：
  - 写指针（wptr）在写时钟域产生 → Gray code 编码 → 两级 FF 同步到读时钟域 → 用于判空
  - 读指针（rptr）在读时钟域产生 → Gray code 编码 → 两级 FF 同步到写时钟域 → 用于判满
  - 同步延迟导致的保守判断：可能假满/假空，但不会真溢出/真下溢

**Block 3：乱序 CPU 复习 + GPGPU 巩固（1h）**

- [ ] 快速过一遍 Day 3 的乱序笔记，确保每个概念都能口述
- [ ] GPGPU 补充：
  - **warp divergence**：同一 warp 内线程走不同分支 → 串行执行 → 效率下降
  - **bank conflict**：多个线程访问 shared memory 同一 bank → 串行化
  - **coalesced memory access**：相邻线程访问相邻地址 → 合并为一次 DRAM transaction
- [ ] 思考题："如果让你设计一个 AI 芯片，你会怎么考虑？"

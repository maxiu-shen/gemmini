## 2. 本周工作内容和进展

### 2.1 参考架构选型调整：从 Eyeriss 转向 Gemmini

经过深入调研和对比分析，本周正式决定将 RSNCPU 项目中脉动阵列部分的**参考架构从 Eyeriss 切换至 Gemmini**。核心理由如下：

#### （1）目标对齐度

RSNCPU（以及其基础 SNCPU）的核心关注点是"**CPU + 脉动阵列的协同与融合**"，即统一的 2D PE 阵列在不同模式下扮演 CPU pipeline 或典型 CNN 脉动阵列的角色，目标是端到端性能提升与 CPU/NN 融合。

- **Gemmini** 的设计目标是为 RISC-V SoC 提供一个可配置的 GEMM 脉动阵列内核，强调**矩阵乘/NN 核心算子与 CPU 的协同**——这与 SNCPU 属于同一类问题。
- **Eyeriss** 的核心创新在于 Row-Stationary（RS）CNN 数据流和多播 NoC，关注的是"如何让 CNN 更省能"，属于另一类问题。

#### （2）阵列形态匹配

SNCPU 论文中的加速模式采用典型 systolic dataflow（WS/OS 风格），即规则 2D MAC 阵列，数据沿行/列按拍流动。

- **Gemmini** 的核心就是一个参数化的 2D systolic array，支持 output-stationary / weight-stationary 等经典数据流，非常贴近 SNCPU 的"典型脉动 MAC 阵列"。
- **Eyeriss v1** 并非传统意义上的"纯 systolic array"，它依赖复杂的多播 NoC + RS dataflow，更像一个为卷积特化的"阵列 + NoC"系统。

#### （3）与 CPU 的耦合方式

- **Gemmini** 原生集成在 Chipyard 生态中，通过 RoCC 接口挂在 Rocket/BOOM 核上，是一个**指令级可控的加速器**，已经解决了 CPU 向阵列下发指令、数据在 CPU 缓存/阵列 Scratchpad 之间搬运等关键问题——这些问题与 SNCPU 关心的"CPU—阵列协同"高度重合。
- **Eyeriss** 更像一个独立 NPU，CPU 在外面喂配置和数据，架构假设是"CPU 和加速器是两个世界"，这与 SNCPU 的"统一阵列"思路是相反的。

#### （4）工程可行性

- **Gemmini** 完整开源（Chisel/Scala）、社区活跃、文档和例程丰富。可以裁剪出 systolic array + 必要控制 + Scratchpad/Accumulator，快速在仿真/FPGA 上跑通矩阵乘或简单网络层。
- **Eyeriss** 官方 RTL 未开源，仅有第三方复现版本，质量和细节难以保证。NoC、RS 数据流、Buffer 层级等工程复杂度高，理解和验证成本大。

#### （5）未来演进空间

从 Gemmini 出发，可以一步步实现：改 RoCC 接口 → 更紧耦合 → 调整 memory 路径 → 减少数据搬运 → 将部分控制逻辑向 CPU pipeline 靠拢，逐步演化出 SNCPU-like 的统一阵列架构。而从 Eyeriss 出发，需要先拆掉大量为 CNN RS dataflow 设计的专有结构，再变成通用 systolic 再谈 CPU 融合，相当于绕了大圈。

> **一句话总结**：Gemmini 在架构目标、阵列形态、与 RISC-V CPU 的协同方式以及开源工程成熟度上，都与 SNCPU 的"典型 systolic 阵列 + CPU/NN 融合"思路高度一致；而 Eyeriss 的核心创新是 CNN Row-Stationary 数据流和 NoC，会把精力从"复现 SNCPU 的阵列与融合思想"转移到"复现特定 CNN 数据流与网络结构"，不利于当前的工程目标。

### 2.2 Gemmini 简介

**Gemmini** 是由 UC Berkeley 开发的一个全系统、全栈 DNN 硬件探索与评估平台（DAC 2021），使用 Chisel 硬件描述语言实现，是 Chipyard 生态的一部分。其核心特点包括：

| 特性 | 描述 |
|------|------|
| **加速器类型** | 基于 RoCC 接口的 RISC-V 自定义指令加速器，挂载在 Rocket/BOOM 核上 |
| **核心计算单元** | 参数化 2D 脉动阵列（默认 16×16），支持 WS 和 OS 两种数据流 |
| **片上存储** | 显式管理的 Scratchpad（分组 SRAM）+ Accumulator（带加法器） |
| **数据搬移** | DMA 引擎在主存（L2 Cache）和 Scratchpad 之间传输数据 |
| **控制架构** | 解耦访问/执行（Decoupled Access/Execute），包含 LoadController、StoreController、ExecuteController 三个独立控制器 |
| **ISA** | 丰富的自定义指令集：`mvin`/`mvout`（数据搬移）、`config_ex`/`config_mvin`/`config_mvout`（配置）、`matmul.preload`+`matmul.compute`（矩阵乘）、CISC 循环指令（自动分块与双缓冲） |
| **冒险检测** | ROB（重排序缓冲区）检测不同控制器之间的指令依赖 |
| **软件支持** | 提供 C 库（`gemmini.h`）封装自定义指令为常见 DNN 算子，支持 ONNX-Runtime |

Gemmini 的高层架构如下：CPU（Rocket Core）通过 RoCC 接口向 Gemmini 加速器发送自定义指令，加速器内部由三个解耦控制器分别处理加载、存储和执行操作，共享 Scratchpad 和 Accumulator SRAM。DMA 引擎负责在主存和片上 SRAM 之间搬运数据。

### 2.3 Gemmini 环境搭建与验证

#### 搭建流程

Gemmini 的搭建基于 Chipyard 生态，按照 [Gemmini 官方 README](https://github.com/ucb-bar/gemmini) 和 [Chipyard 官方文档](https://chipyard.readthedocs.io/en/latest/) 的指引完成，步骤概述如下：

**Step 1：安装系统依赖**

在 Ubuntu 系统上安装 Chipyard 所需的基础依赖，包括 JDK、SBT、Verilator、Python3、Git 等。具体依赖列表参见 [Chipyard 官方文档](https://chipyard.readthedocs.io/en/latest/Chipyard-Basics/Initial-Repo-Setup.html#default-requirements-installation)。

**Step 2：克隆 Chipyard 并初始化**

```bash
git clone https://github.com/ucb-bar/chipyard.git
cd chipyard
./build-setup.sh    # 自动拉取子模块并编译 RISC-V 工具链和 Spike
source env.sh       # 加载环境变量
```

> 注意：`build-setup.sh` 执行时间较长（数小时），主要耗时在 RISC-V GNU 工具链的编译。网络环境可能影响子模块下载速度，建议使用代理或镜像源。

**Step 3：编译 Gemmini Spike 扩展（功能仿真器）**

```bash
cd generators/gemmini
make -C software/libgemmini install
```

此步骤编译 Spike 的 Gemmini 扩展模块（libgemmini），使 Spike 能够识别和执行 Gemmini 自定义指令。

**Step 4：编译 Gemmini 裸机测试程序**

```bash
cd software/gemmini-rocc-tests
./build.sh
```

编译完成后，在 `build/bareMetalC/` 目录下生成裸机二进制文件（如 `mvin_mvout-baremetal`、`matmul-baremetal`、`template-baremetal` 等）。

**Step 5：使用 Spike 运行裸机测试**

```bash
cd build/bareMetalC
spike --extension=gemmini mvin_mvout-baremetal    # 数据搬入搬出测试
spike --extension=gemmini matmul-baremetal         # 矩阵乘法测试
```

#### 验证结果

以下官方裸机测试程序在 Spike 功能仿真器上均通过验证：

| 测试程序 | 功能描述 | 结果 |
|---------|---------|------|
| `mvin_mvout` | 将矩阵从主存搬入 Scratchpad 后再搬回主存，验证数据一致性 | ✅ 通过 |
| `matmul` | 执行基本矩阵乘法运算（使用 `matmul.preload` + `matmul.compute` 指令序列） | ✅ 通过 |
| `template` | 官方模板测试程序 | ✅ 通过 |


> 注：Spike 仅验证功能正确性，不提供精确的性能指标。如需周期级精确仿真，需使用 Verilator 或 VCS 构建 RTL 仿真器（`make CONFIG=GemminiRocketConfig`），编译时间更长。

#### 搭建过程中遇到的问题

1. **子模块下载缓慢**：由于 Chipyard 依赖大量 GitHub 子模块，网络环境不稳定时下载容易中断。解决方案：使用代理或分步手动克隆关键子模块。
2. **工具链编译耗时长**：`build-setup.sh` 中 RISC-V 工具链编译耗时约 2-3 小时（视机器配置），期间占用大量 CPU 和内存资源。
3. **环境变量配置**：每次新开终端需要 `source env.sh` 加载 Chipyard 环境变量，否则 Spike 和交叉编译工具链无法正确调用。
